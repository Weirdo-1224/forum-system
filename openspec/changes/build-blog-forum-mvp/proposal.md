## 为什么

### 当前问题

当前仓库只有 Spring Boot 基础骨架，尚未形成可用的博客论坛后端；认证授权、用户资料、文章发布、分类标签、评论、点赞和管理员审核等核心能力均缺失，现有持久化依赖也需要与项目约定的 MyBatis-Plus、PostgreSQL 和 Flyway 基线对齐。若不先建立统一的 MVP 范围、权限边界和状态规则，后续实现容易出现模块耦合、越权访问、状态冲突和数据库一致性问题。

### 项目目标

从零构建一个可独立部署和逐步验收的 Spring Boot 博客论坛 REST API 后端。系统采用模块化单体架构，为访客、注册用户和管理员提供边界清晰的内容发布与治理能力，并以数据库约束、事务和自动化测试保障安全性、一致性及可维护性。

### 用户类型

- **访客**：可以注册、登录，浏览已发布且可见的文章，按分类和标签筛选文章，并查看公开评论和公开用户主页。
- **注册用户**：除访客能力外，可以创建草稿，编辑、发布和删除自己的文章；评论、回复、编辑和删除自己的评论；点赞和取消点赞文章；维护个人基础资料。发布后的文章允许直接编辑，不触发重新审核。
- **管理员**：除注册用户能力外，可以维护分类，隐藏和恢复文章或评论，禁用和恢复用户，并查看审核操作记录。用户被禁用后历史公开内容继续展示。

## 变更内容

### MVP 范围

- 建立基于 JWT 的注册、登录、Refresh Token 轮换和退出登录流程；不做邮箱验证，Access Token 到期前不强制立即失效。
- 建立用户基础资料和公开主页；公开主页展示基础信息及该用户已发布、未隐藏、未删除的文章。
- 建立文章草稿、编辑、发布、作者删除和公开浏览能力；公开 URL 使用稳定且唯一的 slug，正文使用 Markdown 并经过安全过滤。
- 建立管理员维护分类、用户自由输入并由系统自动创建标签、按分类和标签筛选文章的能力。
- 建立单表扁平存储的两层评论与回复能力；父评论删除后子评论保留并标记为 orphan。
- 建立仅针对文章的幂等点赞和取消点赞能力，并处理重复请求和并发写入。
- 建立文章和评论的管理员隐藏、恢复与审核记录能力；被隐藏内容的作者可查看其状态，其他普通访问者不可见。
- 建立管理员禁用和恢复用户的能力；禁用后拒绝新的登录和 Token 刷新，但不隐藏历史公开内容。
- 统一分页、参数校验、ProblemDetail 风格错误响应、稳定错误码、权限校验和审计边界。

### 明确的非目标

- 不开发前端页面。
- 不支持图片或其他附件上传。
- 不支持私信、用户关注和通知。
- 不支持全文搜索和推荐算法。
- 不引入 Redis、消息队列或其他非必要中间件。
- 不拆分微服务，不引入分布式事务。
- 不做第三方登录、多因素认证、邮箱验证、找回密码或用户自助注销。
- 不支持评论点赞、文章版本历史、协同编辑、定时发布或已删除内容的用户自助恢复。

### 受影响的业务领域

- `auth`：身份认证、Token 生命周期和会话撤销。
- `user`：用户资料、公开主页、角色及账号启停状态。
- `post`：文章生命周期、slug、Markdown、分类、标签和公开查询。
- `comment`：两层评论、回复、作者编辑删除和 orphan 语义。
- `reaction`：文章点赞、取消点赞、幂等和并发一致性。
- `moderation`：内容隐藏恢复、用户启停和审核操作记录。
- `common`：错误响应、分页、校验、时间、ID 和通用安全约定；该模块不承载业务规则。

### 功能优先级

- **P0——可运行核心**：工程与数据库基线、统一错误处理、认证授权、用户状态、文章草稿与发布、公开文章查询。
- **P1——论坛闭环**：公开用户主页、分类标签筛选、两层评论、文章点赞、内容审核和审核记录。
- **P2——发布加固**：权限矩阵回归、并发冲突验证、Markdown 安全过滤、索引与查询优化、OpenAPI、Actuator 和容器化验收。

### 主要风险

- Refresh Token 并发轮换可能产生多个有效会话，重放旧 Token 可能扩大账号风险。
- 所有权校验不完整可能导致用户编辑或删除他人内容。
- 管理员隐藏、作者删除和发布状态混用可能造成内容被错误恢复或泄露。
- MyBatis 动态 SQL、Markdown 渲染和日志可能分别引入 SQL 注入、XSS 和凭据泄露风险。
- slug、用户名、邮箱、标签和点赞在并发或软删除场景下可能产生冲突或身份混淆。
- 父评论删除与新增回复并发时，子评论的 orphan 状态可能不一致。
- 用户禁用后，已签发 Access Token 在最长 30 分钟内仍可能继续访问，属于已接受但必须记录的风险。
- 发布后直接编辑不重新审核，作者可能在内容恢复后再次写入违规内容。

### 分阶段交付方式

1. **基础阶段**：对齐 Spring Boot、MyBatis-Plus、PostgreSQL、Flyway、错误模型、测试容器和模块边界，验收应用启动及数据库迁移。
2. **身份阶段**：交付注册、登录、Token 刷新、退出、角色、用户资料和账号启停，验收完整认证生命周期与权限拒绝场景。
3. **内容阶段**：交付文章草稿、slug、Markdown、发布编辑删除、公开主页、分类标签和分页浏览，验收文章状态及所有权规则。
4. **互动阶段**：交付两层评论、orphan 处理和文章点赞，验收幂等、层级限制和并发冲突。
5. **治理阶段**：交付文章评论隐藏恢复、审核记录及用户恢复，验收作者删除与管理员审核的独立语义。
6. **加固阶段**：完成安全、事务、并发、索引、API 文档、运行监控和端到端回归验收。

### 回滚和失败处理思路

- 每个阶段必须保持可独立部署和验收；后续阶段失败时，回退应用版本不得破坏已完成阶段的公开契约。
- 数据库变化全部通过 Flyway 增量迁移交付；优先采用向前兼容和前向修复，不依赖生产环境手工改表或 Hibernate 自动建表。
- 破坏性数据库调整采用“先扩展、再迁移、后收缩”，在确认旧版本不再依赖后才移除旧结构。
- Token 轮换、点赞、评论删除与审核操作必须在明确事务边界内执行；失败时整体回滚，重复请求应返回稳定且可预测的结果。
- 外部请求失败使用统一错误响应和稳定错误码，不返回部分成功状态，不泄露内部异常、Token 或敏感用户信息。
- 功能上线后若出现安全或一致性问题，优先通过禁用相关写入口、回退应用和执行前向数据修复控制影响；已写入的审核记录不得回滚删除。

## 功能 (Capabilities)

### 新增功能

- `identity-access`: 用户注册、登录、JWT 与 Refresh Token 生命周期、退出登录、角色权限以及管理员禁用和恢复用户。
- `user-profiles`: 用户维护基础资料以及访客查看公开用户主页和该用户公开文章列表。
- `post-publishing`: 文章草稿、稳定 slug、Markdown 安全内容、编辑、发布、作者删除和公开浏览。
- `content-taxonomy`: 管理员维护分类、系统自动创建标签以及按分类和标签筛选文章。
- `two-level-comments`: 两层扁平评论、回复、作者编辑删除、父删子留和 orphan 状态。
- `post-reactions`: 文章点赞、取消点赞、幂等响应和并发唯一性。
- `content-moderation`: 管理员隐藏与恢复文章和评论、作者可见的隐藏状态以及审核操作记录。

### 修改功能

无。当前项目尚无既有业务规范，本变更引入的能力均为新增功能。

## 影响

- **代码与架构**：新增 `auth`、`user`、`post`、`comment`、`reaction`、`moderation` 业务模块，并限制跨模块依赖；`common` 仅提供稳定的横切能力。
- **API**：新增 `/api/v1` 下的认证、用户、文章、分类、标签、评论、点赞和管理端 REST API，并统一分页、校验与错误语义。
- **数据与迁移**：新增 PostgreSQL 业务模型、唯一约束、外键、状态约束和查询索引，所有结构变化由 Flyway 管理。
- **依赖**：持久化基线从当前 JPA 依赖调整为 MyBatis-Plus，并使用 Spring Security、JWT、PostgreSQL、Testcontainers 和 Markdown 安全处理能力。
- **运行与安全**：新增环境化 JWT 密钥、认证过滤、权限控制、日志脱敏、审核审计、健康检查和容器化本地运行约定。
- **兼容性**：当前无已发布业务 API，因此不包含面向既有客户端的破坏性行为；后续阶段必须维持已验收契约的向后兼容。
