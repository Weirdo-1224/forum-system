## 新增需求

### 需求:维护个人资料
系统必须允许注册用户查看和修改自己的基础公开资料，并拒绝非法参数和对他人资料的越权修改。

#### 场景:更新个人资料成功
- **当** 已认证用户提交合法的显示名称、简介或头像地址
- **则** 系统更新该用户的基础资料
- **并且** 后续公开查询返回更新后的公开字段

#### 场景:资料参数非法
- **当** 已认证用户提交超过长度限制、格式非法或包含不允许内容的资料
- **则** 系统拒绝更新并返回参数错误
- **并且** 原有资料保持不变

#### 场景:修改他人资料
- **当** 注册用户尝试修改其他用户的资料
- **则** 系统拒绝操作并返回无权限结果
- **并且** 目标用户资料保持不变

### 需求:公开用户主页
系统必须向访客展示用户的基础公开信息和公开文章列表，并禁止泄露邮箱、账号状态和认证相关信息。

#### 场景:查看公开主页
- **当** 访客请求存在用户的公开主页
- **则** 系统返回该用户的基础公开信息
- **并且** 响应不包含邮箱、Token、内部权限或其他私密信息

#### 场景:用户不存在
- **当** 访客请求不存在或已软删除用户的公开主页
- **则** 系统返回数据不存在
- **并且** 系统不返回任何历史私密资料

#### 场景:查看禁用用户主页
- **当** 访客请求已被管理员禁用但未软删除用户的公开主页
- **则** 系统继续返回该用户的基础公开信息
- **并且** 系统不公开该用户的禁用状态

### 需求:公开主页文章分页
系统必须对公开用户主页中的文章列表提供分页和稳定排序，并仅展示已发布、未隐藏且未删除的文章。

#### 场景:默认分页
- **当** 访客未指定分页和排序参数查看用户文章列表
- **则** 系统按发布时间倒序返回第一页且默认最多 20 条
- **并且** 相同发布时间的文章仍保持稳定顺序

#### 场景:合法分页和排序
- **当** 访客提交合法页码、每页数量和允许的排序方式
- **则** 系统返回对应范围和顺序的公开文章
- **并且** 每页数量不超过 100 条

#### 场景:分页或排序参数非法
- **当** 访客提交负数页码、超过上限的每页数量或不支持的排序字段
- **则** 系统拒绝请求并返回参数错误
- **并且** 系统不执行不受支持的排序表达式

#### 场景:存在非公开文章
- **当** 用户同时拥有草稿、隐藏、删除和正常公开文章
- **则** 系统仅返回正常公开文章
- **并且** 列表总数不包含其他状态的文章

### 需求:用户身份历史一致性
系统必须以不可变身份关联用户历史内容，即使用户名或邮箱在软删除后被新账号复用，也不得将历史内容归属于新账号。

#### 场景:标识被新账号复用
- **当** 软删除用户的用户名或邮箱被新账号合法复用
- **则** 系统将新账号视为独立用户
- **并且** 旧账号的历史文章、评论和审核记录不归属于新账号

#### 场景:禁用用户历史内容
- **当** 管理员禁用一个仍有公开内容的用户
- **则** 系统继续按原作者身份展示其历史公开内容
- **并且** 系统不因账号禁用自动隐藏或删除内容
