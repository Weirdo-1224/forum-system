## 新增需求

### 需求:用户注册
系统必须允许访客使用合法且当前可用的用户名、邮箱和密码注册账号，不要求邮箱验证，并拒绝重复或非法注册信息。

#### 场景:注册成功
- **当** 访客提交合法且当前未被使用的用户名、邮箱和密码
- **则** 系统创建可登录的注册用户账号
- **并且** 系统不在响应中返回密码或其他认证秘密

#### 场景:注册参数非法
- **当** 访客提交格式非法的用户名、邮箱或不满足安全要求的密码
- **则** 系统拒绝注册并返回参数错误
- **并且** 系统不创建任何用户账号

#### 场景:有效账号标识重复
- **当** 访客提交已被未软删除账号占用的用户名或邮箱
- **则** 系统拒绝注册并返回标识冲突
- **并且** 系统不泄露该标识对应账号的其他信息

#### 场景:软删除标识复用
- **当** 访客提交仅被软删除账号使用过且当前无人占用的用户名或邮箱
- **则** 系统允许使用该标识完成注册
- **并且** 新账号与历史账号保持不同身份

### 需求:用户登录
系统必须允许有效账号凭正确凭据登录，并对错误凭据、禁用账号和不存在账号返回一致且不泄露账号状态的失败结果。

#### 场景:登录成功
- **当** 有效用户提交正确的登录标识和密码
- **则** 系统签发有效期为 30 分钟的 Access Token 和有效期为 7 天的 Refresh Token
- **并且** 系统返回当前用户的必要身份与权限信息

#### 场景:凭据错误或账号不存在
- **当** 访客提交错误密码或不存在的登录标识
- **则** 系统拒绝登录并返回统一的认证失败结果
- **并且** 系统不说明账号是否存在

#### 场景:禁用账号登录
- **当** 已被管理员禁用的用户提交正确凭据
- **则** 系统拒绝登录
- **并且** 系统不签发任何新 Token

### 需求:受保护资源认证
系统必须要求调用受保护资源的请求携带有效 Access Token，并依据用户身份和角色执行权限判断。

#### 场景:有效 Token 访问
- **当** 用户携带未过期且有效的 Access Token 调用其有权限的操作
- **则** 系统允许请求继续处理
- **并且** 系统以 Token 对应用户作为当前操作者

#### 场景:缺少或非法 Token
- **当** 请求未携带 Access Token，或 Token 已过期、签名非法、格式非法
- **则** 系统拒绝访问并返回认证失败
- **并且** 系统不执行受保护操作

#### 场景:角色权限不足
- **当** 已认证用户调用仅管理员可执行的操作
- **则** 系统拒绝访问并返回无权限结果
- **并且** 系统不改变任何业务状态

### 需求:Refresh Token 轮换
系统必须在每次成功刷新时使旧 Refresh Token 失效并签发一组新 Token，且同一个旧 Token 在并发请求中最多成功一次。

#### 场景:刷新成功
- **当** 用户提交未过期、未撤销且尚未被轮换的 Refresh Token
- **则** 系统使该 Refresh Token 失效并签发新的 Access Token 和 Refresh Token
- **并且** 新旧 Token 的有效状态切换作为一个不可分割的结果完成

#### 场景:Refresh Token 过期或非法
- **当** 用户提交过期、格式非法或无法识别的 Refresh Token
- **则** 系统拒绝刷新并返回认证失败
- **并且** 系统不签发任何新 Token

#### 场景:并发刷新
- **当** 两个请求并发使用同一个有效 Refresh Token
- **则** 系统只允许其中一个请求完成轮换
- **并且** 其他请求收到该 Token 已失效的失败结果

#### 场景:重放已轮换 Token
- **当** 用户再次提交已经成功轮换过的 Refresh Token
- **则** 系统拒绝刷新并将其识别为 Token 重放
- **并且** 系统撤销同一登录会话中仍有效的 Refresh Token

### 需求:退出登录
系统必须允许已认证用户撤销当前登录会话的 Refresh Token，且重复退出请求保持幂等。

#### 场景:退出成功
- **当** 用户提交当前会话的有效 Refresh Token 请求退出
- **则** 系统撤销该 Refresh Token
- **并且** 后续使用该 Token 的刷新请求均被拒绝

#### 场景:重复退出
- **当** 用户对已经退出或已经撤销的同一会话再次请求退出
- **则** 系统返回与退出完成一致的结果
- **并且** 系统不产生额外会话或状态变化

#### 场景:退出参数非法
- **当** 用户提交格式非法且无法识别的 Refresh Token 请求退出
- **则** 系统拒绝请求并返回参数或认证错误
- **并且** 系统不影响其他登录会话

### 需求:禁用账号的认证生命周期
系统必须阻止禁用账号进行新登录和 Token 刷新，但允许已签发的 Access Token 在原定有效期内自然失效。

#### 场景:禁用后刷新
- **当** 已被禁用的用户使用此前有效的 Refresh Token 请求刷新
- **则** 系统拒绝刷新
- **并且** 系统不签发新的 Token

#### 场景:已有 Access Token 自然失效
- **当** 用户被禁用时仍持有尚未到期的 Access Token
- **则** 系统不要求该 Access Token 立即失效
- **并且** Token 到达原定过期时间后不得继续使用
